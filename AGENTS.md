# AGENTS.md

本文件定义本仓库内所有 agent 的统一工作约定。除非任务明确要求，否则默认按本文件执行。

## 项目概览

- 项目名称：`WuWaChat`
- 技术栈：Electron 42、React 19、TypeScript 5、electron-vite、Tailwind CSS 4
- 项目定位：桌面端角色聊天应用，涵盖角色会话、模型配置、记忆检索与 embedding 工作流

## 仓库结构

- `src/main`：Electron 主进程，负责应用生命周期、窗口、IPC、AI 运行时、设置、记忆与本地服务
- `src/preload`：预加载桥接层，向渲染进程暴露受控 API
- `src/renderer`：React 界面、交互逻辑、hooks、Zustand 状态管理、样式
- `src/shared`：跨进程共享的类型、配置和轻量契约
- `resources`：打包资源
- `docs`：项目文档与截图
- `out`：构建产物，禁止手工修改

## 常用命令

- 安装依赖：`npm install`
- 启动开发环境：`npm run dev`
- 代码检查：`npm run lint`
- 类型检查：`npm run typecheck`
- 构建应用：`npm run build`
- 平台打包：
  - Windows：`npm run build:win`
  - macOS：`npm run build:mac`
  - Linux：`npm run build:linux`

## 架构边界

- 必须显式维护进程边界，不允许模糊主进程、预加载层和渲染层职责。
- Electron 生命周期、文件系统访问、模型运行时、IPC 注册与系统能力必须放在 `src/main`。
- 仅通过 `src/preload` 向渲染层暴露原生能力，禁止 renderer 直接依赖主进程实现。
- `src/renderer` 只承担浏览器安全的界面与交互逻辑，不直接接入 Node/Electron 原生能力。
- `src/shared` 只放共享类型、轻量配置与跨进程契约，禁止塞入带副作用的运行时代码。

## 路径别名

- main / preload：`@main/*`、`@shared/*`
- renderer：`@renderer/*`、`@shared/*`
- 文件已使用别名时，后续修改必须延续现有写法，避免退回冗长相对路径。

## 代码风格

- 所有新增代码默认使用 TypeScript，并保持类型清晰可追踪。
- 格式化规则以 Prettier 为准：
  - 单引号
  - 不写分号
  - `printWidth: 100`
  - `trailingComma: none`
- 静态检查规则以现有 ESLint 配置为准，包含 Electron Toolkit、React、React Hooks、React Refresh。
- 命名、模块拆分、目录组织必须优先对齐现有代码，而不是引入个人偏好。
- 注释只用于解释不直观的约束、边界或原因，禁止用注释重复代码表面含义。

## 编码质量要求

- 所有改动必须形成最小闭环，只解决当前任务，禁止顺手做无关重构。
- 新增或修改逻辑时，必须优先复用现有分层、模块和模式；没有明确收益时，不得引入新抽象。
- 禁止无理由使用 `any`、`@ts-ignore`、大范围 `eslint-disable`。确实无法避免时，必须缩小作用范围，并在代码中说明原因。
- 所有函数都必须具备 JSDoc 注释(React 组件除外)，适用范围覆盖 `main`、`preload`、`renderer`、`shared`。
- 函数的 JSDoc 中，`@description` 为必填项，必须准确描述函数职责，禁止留空、禁止写无信息量占位内容。
- `@param`、`@returns`、`@remarks` 等标签按函数实际情况补充，不要求机械套用固定模板；标签内容必须和函数签名、返回值、副作用及使用约束一致。
- 有参数且参数语义、约束、副作用或可空性不直观时，必须补充 `@param`。
- 有返回值且返回语义不直观时，必须补充 `@returns`。
- 有副作用、边界条件、异常行为、跨进程约束或特殊使用前提时，必须补充 `@remarks`。
- 禁止只写注释壳子而没有有效内容，禁止 `@description` 与函数行为不符，禁止复制模板后不按实际签名删改。
- 禁止吞错。`catch` 分支必须执行以下至少一项：记录日志、转换错误、向上抛出；不得静默失败。
- 涉及 `main`、`preload`、`renderer` 的跨进程改动时，必须同步更新共享类型、桥接接口和调用方，禁止只改单侧。
- 涉及 memory、embedding、chat runtime 的改动时，必须同时检查失败路径、日志记录和资源释放，不得只验证成功路径。
- Renderer 层不得绕过 preload 直接访问原生能力、主进程模块或不安全接口。
- 涉及状态管理、异步流程或 IPC 返回值的改动时，必须考虑空状态、加载中、失败回退和重复触发场景。
- 允许例外，但例外必须满足两个条件：
  - 例外范围最小化
  - 在结果说明中明确写出原因、影响和后续风险

## 变更原则

- 先理解现有实现，再决定改动点；禁止未读上下文就直接改文件。
- 修改导出函数的签名或行为时，必须同步更新对应 JSDoc，禁止注释与实现漂移。
- 改动优先保持局部一致性，除非任务要求，否则不重命名、不搬迁、不批量整理无关代码。
- 不得手工修改 `out`、构建产物或无任务需求的锁文件。
- 现有 IPC 契约默认视为稳定接口；除非调用链两端同时更新，否则不得擅自变更。
- 修改设置、记忆、模型或日志相关逻辑时，必须同时检查配置持久化、错误传播和可观测性。

## 验证要求

- 验证是硬性门槛，不是可选项。提交前至少执行与改动范围匹配的检查。
- 默认优先执行：
  - `npm run lint`
  - `npm run typecheck`
- 仅文档改动、纯资源改动或仓库现状明确不需要时，可以不运行上述命令，但必须在结果中说明未执行原因。
- 如果无法完成完整验证，必须明确写出：
  - 未执行的检查项
  - 阻塞原因
  - 可能受影响的范围

## Agent 工作流

1. 先阅读相关代码、类型和边界，确认改动属于哪个层级。
2. 再确定最小改动方案，避免把实现问题扩大成结构性重写。
3. 然后实施改动，并保持跨文件契约一致。
4. 改完后执行匹配的验证，检查主路径与失败路径。
5. 最后汇报变更内容、影响范围、验证结果和剩余风险。

## 实务说明

- 运行时用户数据保存在仓库外部，禁止假设工作区内文件就是应用状态源。
- 本项目同时包含 UI、IPC、本地 AI 运行时与记忆服务，跨层改动通常需要联动检查 main、preload、renderer 与 shared。
- 新功能如果跨进程，默认按“共享类型 -> preload 接口 -> main IPC 实现 -> renderer 调用”的顺序设计和落地。
- 如果后续要把本文件中的质量要求落地为 ESLint 规则、测试基建或 CI 阻断，请单独立项，不在普通任务中顺手扩张范围。
